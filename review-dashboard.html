■ Critical Findings Must Fix

#5 · PAY CALCULATION
OT Rate Formula is Wrong
mobile/data/mockData.ts:205
Critical
The OT rate computes (Base + Differential) × 1.5 but the BCMEA contract specifies (Base × 1.5) + Differential (flat). Every job with a differential shows incorrect pay estimates. For a Class 1 trade like HD MECHANIC (+$2.50/hr), this overestimates OT by $1.25/hr.
// Current (WRONG): const otRate = Math.round(regRate * 1.5 * 100) / 100 // Correct per BCMEA contract: const otRate = Math.round((baseRate * 1.5 + differential) * 100) / 100
⚠ Affects every user with a differential job — wrong pay estimates on 957 active users
#1 · TIMEZONE BUG
calculateWeeklyEarnings uses new Date() on date strings
mobile/data/mockData.ts:251
Critical
Your own formatters.ts header says: “Date strings are compared lexicographically (no new Date(dateString))” — but calculateWeeklyEarnings violates this rule. new Date('2026-03-19') parses as UTC midnight, which is 4-8 PM the previous day in Pacific time. Shifts at week boundaries get miscounted.
// Current (WRONG — timezone-unsafe): .filter(s => { const shiftDate = new Date(s.date) return shiftDate >= startOfWeek && shiftDate < endOfWeek }) // Correct (matches calculateYTDEarnings pattern): .filter(s => { const d = s.date.slice(0, 10) return d >= weekStartStr && d < weekEndStr })
⚠ Affects every user's home screen earnings display at week boundaries
#4 · DATA TRUNCATION
Shift fetch hard-limited to 200 rows
mobile/hooks/useShifts.ts:80
Critical
Power users logging 5 shifts/week hit 200 in ~9 months. After that, their pension YTD calculation, analytics, and calendar silently miss older shifts. The pension progress bar — the core value prop — would show incorrect progress. Your most engaged users break first.
const { data, error } = await supabase .from('shifts') .select('*') .eq('user_id', user.id) .order('date', { ascending: false }) .limit(200) // ← 40 weeks of daily shifts
⚠ Will silently break for your most engaged users first — 9 months from launch
